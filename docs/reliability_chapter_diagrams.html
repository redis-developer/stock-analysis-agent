Reliability patterns for agentic applications

The stock analysis chatbot becomes a durable workflow. Redis stores the state required to observe, recover, coordinate, and resume agent work.

Durable stateWorkflow hashes, status, TTL leases, and session links.
Replayable historyStreams capture what the agent did and what it saw.
Coordinated recoveryLocks, idempotency, queues, and checkpoints keep replicas safe.
Controlled executionCircuit breakers, semaphores, DLQs, and human approval.

Chapter 1

A chat request becomes a workflow

What if we need to observe, debug, or recover one chat request after it leaves the HTTP handler?
Implementation
Workflow identity
  • Create a workflow id before the agent starts.
  • Store user, session, status, owner, attempt, and timestamps.
  • Connect events, checkpoints, Grafana, and UI progress to the same id.

The workflow becomes visible

What if we need to see what the agent actually did?
Implementation
Redis Streams event log
  • Append ordered events for every agent step.

  • Use the same stream for UI and Grafana inspection.

Chapter 3

Completed work becomes reusable

What if the agent finishes a valuable step and fails later?
Implementation
Checkpoints
  • Promote completed steps to replay points.
  • Store input, output, actor, step id, and summary.
  • Keep checkpoints separate from the full event log.

Chapter 4

Recovery starts from the last checkpoint

What if the process crashes after expensive work already happened?
Implementation
Checkpoint replay
  • Read the latest checkpoint and later events.
  • Send recovered evidence to the coordinator.
  • Create a new workflow in the same conversation.

Blocked work waits outside the chat request

What if the agent reaches a red light and the browser should not wait?
Implementation
Work queue
  • Return to the user with a waiting state.
  • Put the workflow id into Redis.
  • Resume when capacity or provider health returns.

Chapter 6

Distributed workers process the queue safely

What if two replicas pick up the same workflow?
Implementation
Idempotent step execution and distributed locks
  • Acquire a Redis lock before execution.
  • Mark the original workflow hash as recovered.
  • Duplicate workers read the recovered workflow id.

Implementation
Circuit breaker

Chapter 8

Provider capacity is protected

What if the provider is healthy but can only handle a few concurrent calls?
Implementation
Distributed semaphore
  • Acquire a permit before the provider call.
  • Share the active count across replicas.
  • Use TTLs to release leaked permits after crashes.
Redis protects provider concurrency
